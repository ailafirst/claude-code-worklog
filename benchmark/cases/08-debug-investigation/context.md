**事故（prod-incident-0630）**：webhook 处理延迟在 **14:00 突然涨了 ~10 倍**，
P99 从 ~80ms 飙到 ~800ms，一直没回落。

今天的调查（都是排除法，没改业务代码）：

- **排除 DB**：查了慢日志和 DB 端 query 耗时，14:00 前后**平稳**，不是数据库。
- **排除外部支付商**：对方 status page 全绿，我方调用它们的耗时也没变——不是上游。
- **找到强相关**：延迟尖峰的时间点**和 14:00 的一次部署完全重合**。那次部署把
  worker 并发从 **8 调到了 16**。

**当前假设（未证实）**：DB 连接池 `max=10`。并发 16 > 池上限 10，于是多出的 worker 在
**抢连接（pool checkout 排队）**，排队时间表现为处理延迟。8 个 worker 时不触顶，所以之前没事。

为什么还没结论：没有直接证据，只有时间相关性。工作区加了一行 `log webhook took ...ms`
想拿到分段耗时，还没收集够样本。

**验证计划（下一步，二选一就能定性）**：
1. staging 把连接池 `max` 提到 20，回放生产流量，看延迟是否回落；或
2. 直接把并发**回滚到 8**，确认线上延迟掉回 80ms——能证实就是并发/池争用。
